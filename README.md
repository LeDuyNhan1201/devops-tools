# devops-tools

## Project overview

Local DevOps stack and supporting configs for development workflows.

The validated local flow currently starts:
- `envoy`
- `postgres`
- `keycloak0`
- `keycloak1`
- `keycloak2`

Kafka, Logstash, and observability assets exist in the repo, but they are not part of `deployment/docker-compose.local.yml`.

## Structure

- `deployment/`: Makefile and local Docker Compose file
- `scripts/local/`: init, start, stop, clean, rebuild, and certificate helpers
- `services/`: service Dockerfiles and shared service assets
- `environments/local/`: local-only runtime configs, imports, exports, and generated Kafka config targets
- `observability/`: Grafana, Loki, Tempo, Prometheus, and OpenTelemetry Collector configs
- Generated at runtime: `.env`, `secrets/`, `keypair/`, `data/`

## Prerequisites

- Docker and Docker Compose
- `bash`
- `make`
- `openssl`
- `envsubst`
- `keytool`
- `sudo` access

`make prepare` uses `apt-get` and installs `make` and `openjdk-17-jdk`.

## Setup

From the repo root:

```bash
make -f deployment/Makefile prepare
make -f deployment/Makefile init
```

`init` does the following:
- generates `.env`
- generates Kafka config files from templates
- creates secrets, certificates, JWT keypair, and data folders
- builds local Postgres and Keycloak images
- installs the generated CA certificate into the system trust store

## Run locally

Start the validated local stack:

```bash
make -f deployment/Makefile start
docker compose --env-file ./.env -f ./deployment/docker-compose.local.yml ps
```

Stop it:

```bash
make -f deployment/Makefile stop
```

Useful ports from the current Compose file:
- gateway TLS: `8889`
- Postgres: `5438`
- Keycloak nodes: `8881`, `8882`, `8883`
- Keycloak management: `9991`, `9992`, `9993`

Direct script entrypoints:

```bash
./scripts/local/init.sh dev
./scripts/local/start.sh dev
./scripts/local/stop.sh dev
```

## Config (.env / environments if exists)

- Source of truth for generated env values: `scripts/local/helper/env_config.sh`
- `.env` is generated by `init` and `start`
- generated secrets currently include:
  - `secrets/postgres/postgres.password`
  - `secrets/kafka/kafka.secret`
- Kafka config files are generated from:
  - `services/kafka/templates/server.template`
  - `services/kafka/templates/client.template`
- Generated Kafka config targets:
  - `environments/local/kafka/configs/server.properties`
  - `environments/local/kafka/configs/client.properties`
- Generated Kafka cert folders now also expose stable `keystore.p12` and `truststore.p12` aliases alongside the service-specific PKCS12 filenames so Kafka mounts and generated configs resolve the same paths.
- The generated Kafka client properties now assume mTLS-capable `SASL_SSL`: clients need both `keystore.p12` and `truststore.p12` mounted at `/etc/kafka/secrets/` when connecting to brokers with `KAFKA_SSL_CLIENT_AUTH='required'`.
- Local runtime configs live under `environments/local/`:
  - `envoy/envoy.yaml`
  - `postgres/config/` and `postgres/init-db/`
  - `keycloak/import/` and `keycloak/export/`

If you need to change namespace, image tags, hostnames, ports, or bootstrap credentials, edit `scripts/local/helper/env_config.sh` and regenerate with `init`.

If you enable Kafka file-based SSL credential settings in `deployment/kafka.dev.yml` (`*_CREDENTIALS`), make the mounted filename match the generated secret file. The current secret generation flow creates `secrets/kafka/kafka.secret`.

## Troubleshooting

- Run `make -f deployment/Makefile init` before `start`. The validated flow depends on generated certs, images, secrets, keypair, `.env`, and data folders.
- `.env` is regenerated by the scripts. If you edit config inputs, rerun `init`.
- If startup fails on an older branch or partial refactor state, check `scripts/local/start.sh` and `scripts/local/init.sh` first. The recent validation fixed a stale helper call in `start.sh` and an undefined Postgres hostname dependency in `init.sh`.
